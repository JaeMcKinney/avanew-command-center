## Architecture: how RA demo pages render

### Dual-template system (shipped 2026-06-19)

`get_ra_landing_page(p_slug)` v4 returns:
- `template_html` (for `/refer/:slug`)
- `template_demo_html` (for `/demo/:slug`) — THIS is what custom Skilldora work hooks into
- `ra_type` ('individual' | 'company')
- Plus all RA fields (`display_name`, `bio`, `photo_url`, `contact_email`, `partner_company_name`, etc.)

Resolution priority:
1. Per-RA override (`ra_associates.template_id`)
2. Type default (`ra_landing_templates.default_for_type = ra.ra_type`)
3. Legacy org default (`ra_landing_templates.is_default = true`)
4. Frontend builtin fallback (in `src/lib/landingTemplate.ts`)

### Render path for `/demo/:slug`
[src/pages/public/RaDemoPage.tsx](../src/pages/public/RaDemoPage.tsx):
1. `useParams()` reads slug
2. `getRaLandingPage(slug)` → fetches RPC, reads `template_demo_html`
3. If `template_demo_html` is non-empty:
   - Inject `<script>window.__RA_DEMO_SLUG__="slug"</script>` after `<head>`
   - Render as `<iframe srcDoc="...">`
4. If empty:
   - Render `<iframe src="/demo.html?ra=slug">` (static fallback)

This means **any RA without a custom template falls back to `public/demo.html`**. So:
- Editing `public/demo.html` updates the default look for **everyone** (jae, a2ai, future Individuals + Companies that haven't been given a custom)
- Editing the Skilldora template (`public/demo-skilldora.html` via the generator script) only affects Skilldora

### Slug-aware hydration inside the template
The HTML doesn't bake in RA data — it reads `window.__RA_DEMO_SLUG__` and then calls the same `get_ra_landing_page` RPC to populate name/photo/bio/contacts/etc. This is why my Skilldora template is mostly a CSS theme override on top of the canonical structure.

### Things to remember about the generator
- **The base `public/demo.html` hard-codes a LOT of RGBA dark-navy values** that are NOT controlled by CSS custom properties. Every time the user finds a dark-on-dark issue, the fix is to add a new selector override in the THEME_OVERRIDE block.
- **Selectors that DON'T exist in the markup are silent dead code.** I burned an iteration writing `.investment-card` / `.pricing-card` overrides when the actual markup uses `.price.one` / `.price.svc`. Always grep `public/demo.html` for the class name before writing an override.
- **Specificity:** original rules in `<style>` block have NO `!important`. The THEME_OVERRIDE block is appended later in source order so equal-specificity overrides win. Use `!important` defensively anyway — it makes intent clear and beats the original even if it's re-arranged.
- **Decorative elements** (orbs, halos, glows, gold gradients) are designed for a dark backdrop. They look muddy on white — soften with `opacity: 0.25-0.4` rather than removing.
- **Partnership banner** stays dark navy (`linear-gradient(135deg, #122B46, #1c3a63)`) so the partner's white-on-transparent logo reads. Only place the dark theme survives on the Skilldora page.

## Strict scoping — Skilldora changes don't touch other RAs

Three locks ensure isolation:

1. **Template UPDATE is keyed on slug:**
   ```sql
   UPDATE public.ra_associates SET template_id = v_template_id WHERE slug = 'skilldora';
   ```
2. **The template row's `default_for_type` is NULL** — so brand-new Company RAs don't auto-pick it.
3. **Source files are separate.** `public/demo.html` (canonical) is never modified by the Skilldora generator; only `public/demo-skilldora.html` is written. So any edit to `demo.html` propagates to all RAs *except* Skilldora, who is locked to her snapshot.

Live RPC test: `get_ra_landing_page('jae')` returns `template_name: "RA Individual"` with empty `template_demo_html` (falls back to `demo.html`). Only `get_ra_landing_page('skilldora')` returns `template_name: "Skilldora Custom (Company)"`.

1. **Preview screenshot tool sometimes returned blank** — DOM/computed-style verification worked instead. Not blocking. May be a viewport collapse glitch. Resize to fixed dimensions worked once.
2. **Local dev needs `.env`** — `VITE_SUPABASE_URL` + `VITE_SUPABASE_ANON_KEY`. The static `/demo-skilldora.html?ra=...` path works without it (calls Supabase REST directly).
3. **The Skilldora generator depends on specific text in `public/demo.html`** for its REPLACEMENTS list (logo URLs, footer text, title). If those exact strings are edited in `demo.html`, the generator will print `[warn] replacement target not found` — investigate before pushing.
